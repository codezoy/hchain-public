## Human Summary

HCHAIN는 두 레이어로 구성된다. **Core**(`install.sh`)는 어느 프로젝트에든 Harness를 설치하는 프로젝트 독립 엔진이고, **Harness**는 설치된 프로젝트 내에서 실제로 실행되는 AI 멀티에이전트 런타임이다.

워크플로우는 6단계 파이프라인이다: **PLAN → RESEARCH → ACTION → REVIEW → VALIDATE → DONE**. RESEARCH에서 Gemini CLI가 기술 조사를 수행하고, ACTION에서 Claude(사용자)가 코드를 작성하며, REVIEW에서 Codex CLI가 정적 코드 감사를 수행하고, VALIDATE에서 typecheck/lint/test/API health 등 런타임 검증이 이루어진다.

이슈 등급은 CRITICAL > MAJOR > MINOR > NIT이다. 기본적으로 MAJOR 이상 이슈 발견 시 ACTION이 재시도된다. `loop_count`가 3에 달하면 Safety Break가 발동되어 모든 자동 진행이 멈추고 사용자 개입이 필요하다.

태스크는 파일시스템 기반 4-상태 큐로 관리된다(`pending → running → done / blocked`). 각 태스크는 `TASK_YYYYMMDD_NNN` 형식 ID를 가지며 `.md` 정의 파일, `.state.json`, `.checkpoint.json`이 함께 존재한다.

HCHAIN Core 자체는 런타임 상태를 자신의 디렉토리에 기록하지 않는다. 모든 실행 상태와 로그는 설치 대상 프로젝트의 `harness/`와 `.hchain/` 디렉토리에만 저장된다.

## 1. Architecture Overview

```
HCHAIN Core (install.sh)         ← 프로젝트 독립, 재사용 가능
    │
    └── installs into →  <target-project>/
                             ├── harness/           ← Harness Runtime
                             │   ├── harness_runner.sh
                             │   ├── active_state.json
                             │   ├── agents/
                             │   ├── docs/
                             │   ├── lib/
                             │   ├── queue/
                             │   ├── tasks/
                             │   ├── logs/
                             │   └── findings/
                             ├── .hchain/
                             │   └── meta.json      ← 설치 메타데이터
                             └── CLAUDE.md          ← HCHAIN 정책 블록 주입
```

**Core 격리 원칙**: HCHAIN Core는 자신의 디렉토리(`/path/to/hchain/`)에 어떤 런타임 상태 파일도 기록하지 않는다. 모든 실행 상태는 설치 대상 프로젝트 디렉토리에만 저장된다.

### 요구사항

| 컴포넌트 | 요구사항 |
|---|---|
| `install.sh` | bash 3.2+ |
| `harness_runner.sh` 및 harness 전체 | bash 4.0+ |
| 런타임 | `jq` |
| RESEARCH 단계 | `gemini` CLI |
| REVIEW 단계 | `codex` CLI |

macOS에서 bash 4.0+ 설치: `brew install bash`

## 2. Installation

### 기본 설치

```bash
# HCHAIN Core 위치로 이동
cd /path/to/hchain

# 대상 프로젝트에 설치
bash install.sh <target_project_path>

# 또는 --target 플래그 사용
bash install.sh --target /path/to/my-project
```

### 설치 미리보기 (Dry-run)

실제로 파일을 변경하지 않고 변경 사항을 미리 확인한다.

```bash
bash install.sh --target /path/to/my-project --dry-run
```

### 설치 후 변경 사항

설치 완료 시 대상 프로젝트에 다음이 생성된다:

1. **`harness/`** — Harness 전체 디렉토리 구조
2. **`.hchain/meta.json`** — 설치 메타데이터
3. **`CLAUDE.md`** — HCHAIN 정책 블록 주입 (기존 파일이 있으면 append, 없으면 새로 생성)

### meta.json 내용

```json
{
  "hchain_version": "0.1.0",
  "hchain_commit": "abc1234",
  "installed_at": "2026-01-01T00:00:00Z",
  "host": "hostname",
  "hchain_path": "/path/to/hchain",
  "os": {
    "system": "Linux",
    "release": "...",
    "machine": "x86_64"
  }
}
```

### 설치 확인

```bash
bash install.sh --verify /path/to/my-project
```

`.hchain/meta.json`이 존재하면 `installed`, 없으면 `NOT installed`를 출력한다.

### CLAUDE.md 정책 블록

설치 시 `CLAUDE.md`에 다음 블록이 주입된다:

```
<!-- HCHAIN_POLICY_START -->
# HCHAIN Mandatory Execution Policy

이 프로젝트의 모든 코드/문서 수정은 HCHAIN Task를 통해서만 수행한다.

금지:
- TASK.md 없이 코드 수정 금지
- queue/pending 등록 없이 작업 시작 금지
- REVIEWER/VALIDATOR 로그 없이 완료 보고 금지
- 하네스 실행 상태 확인 없이 작업 완료 보고 금지

필수:
- TASK 생성
- queue 등록
- PLAN → RESEARCH → ACTION → REVIEW → VALIDATE → DONE 기록
- logs 생성
- queue/done 이동
- 최종 보고서에 task_id, logs, validation 결과 포함

<!-- HCHAIN_POLICY_END -->
```

기존 블록이 있으면 마커 사이를 교체한다 (idempotent).

## 4. Harness Commands Reference

모든 명령은 대상 프로젝트 루트에서 실행한다.

### 기본 명령

```bash
# 태스크 실행
bash harness/harness_runner.sh --task TASK_20260101_001

# 중단된 태스크 재개
bash harness/harness_runner.sh --resume TASK_20260101_001

# 모든 태스크 목록 조회
bash harness/harness_runner.sh --list

# 특정 태스크 상태 조회
bash harness/harness_runner.sh --status TASK_20260101_001

# Findings 목록 조회
bash harness/harness_runner.sh --findings
```

### Chain 실행

여러 태스크를 순서대로 자동 실행한다.

```bash
# 태스크 범위 실행 (from → to)
bash harness/harness_runner.sh --chain --from TASK_20260101_001 --to TASK_20260101_005

# 선택한 태스크만 실행
bash harness/harness_runner.sh --chain --select TASK_20260101_001,TASK_20260101_003
```

### 플래그

| 플래그 | 설명 |
|---|---|
| `--force` | 이미 running/done 상태인 태스크 강제 재실행 |
| `--dry-run` | 실제 변경 없이 시뮬레이션 (gate 자동 승인) |
| `--override-severity MAJOR\|MINOR\|NIT` | 해당 등급 이하 이슈 무시 (CRITICAL은 항상 유지) |
| `--skip-validate` | VALIDATE 단계 생략 |
| `--no-chain` | DONE 후 다음 pending 태스크 자동 시작 비활성화 |
| `--auto-commit` | DONE 시 git checkpoint 자동 생성 |

### 환경 변수

| 변수 | 기본값 | 설명 |
|---|---|---|
| `HARNESS_AUTO_COMMIT` | — | `1`이면 --auto-commit과 동일 |
| `HARNESS_AUTO_CONFIRM` | — | `1`이면 모든 단계 gate를 자동 승인 |
| `HARNESS_TOKEN_LIMIT` | — | 토큰 한도 초과 시 INTERRUPTED |
| `GEMINI_TIMEOUT` | 900s | Gemini CLI 타임아웃 |
| `CODEX_TIMEOUT` | 1200s | Codex CLI 타임아웃 |
| `VALIDATE_TIMEOUT_DEFAULT` | 600s | 일반 검증 타임아웃 |
| `VALIDATE_TIMEOUT_E2E` | 1800s | E2E 검증 타임아웃 |

### 큐 일관성 검사

```bash
# 기본 검사 (C1: 중복 큐 위치)
bash harness/queue/check_consistency.sh

# 확장 검사 (C1~C8)
bash harness/queue/check_consistency.sh --extended
```

확장 검사 항목:

| 체크 | 설명 |
|---|---|
| C1 | 큐 중복 위치 없음 |
| C2 | stale running 태스크 |
| C2b | blocked 태스크가 wrong queue에 있음 |
| C3 | active_state와 task state 불일치 |
| C4 | state.json 없는 태스크 |
| C5 | orphan running 마커 |
| C6 | step 불일치 |
| C7 | recovery.json 없는 blocked 태스크 |
| C8 | loop_count 불일치 |

### 큐 상태 수동 이동

```bash
bash harness/queue/move.sh TASK_20260101_001 <from> <to>
# from/to 유효값: pending | running | done | blocked
```

## 5. Task Management

### 태스크 파일 구조

각 태스크는 4개 파일로 구성된다:

```
harness/tasks/
├── TASK_20260101_001.md             ← 태스크 정의 (Source of Truth)
├── TASK_20260101_001.state.json     ← 장기 상태 추적
├── TASK_20260101_001.checkpoint.json ← 재개용 컨텍스트
harness/queue/pending/
└── TASK_20260101_001                ← 큐 위치 마커 (빈 파일)
```

### 태스크 정의 파일 (TASK_ID.md)

```markdown
---
task_id: TASK_20260101_001
title: 로그인 API 보안 강화
retry_limit: 3
severity_stop: MAJOR
validate: true
---

# Task: 로그인 API 보안 강화

## Goal
...

## Acceptance Criteria
...

## validate
- type: TYPECHECK
- type: API
  target: http://localhost:8801/health
```

### frontmatter 필드

| 필드 | 타입 | 기본값 | 설명 |
|---|---|---|---|
| `task_id` | string | 파일명에서 유추 | 고유 태스크 ID |
| `title` | string | — | 태스크 제목 |
| `retry_limit` | int | 3 | 최대 ACTION 재시도 횟수 |
| `severity_stop` | CRITICAL\|MAJOR\|MINOR\|NIT | MAJOR | 이 등급 이상 이슈 시 ACTION 재진입 |
| `validate` | bool 또는 목록 | true | VALIDATE 단계 실행 여부 및 검증 항목 |

frontmatter가 없는 파일도 기본값이 적용되어 오류 없이 동작한다 (`task_meta.sh`의 fallback 처리).

### 태스크 ID 규칙

- 형식: `TASK_YYYYMMDD_NNN`
- 예: `TASK_20260101_001`, `TASK_20260515_042`
- frontmatter에 명시하지 않으면 파일명에서 자동 추론

### TASK_ID.state.json 구조 (확인 필요: 실제 스키마는 harness_runner.sh 내부 로직으로 생성)

```json
{
  "task_id": "TASK_20260101_001",
  "status": "running",
  "next_resume_step": "ACTION",
  "loop_count": 1,
  "title": "로그인 API 보안 강화"
}
```

### TASK_ID.checkpoint.json 구조

재개 시 사용되는 컨텍스트를 저장한다.

```json
{
  "task_id": "TASK_20260101_001",
  "resume_prompt": "ACTION 단계 재개: ...",
  "next_resume_step": "ACTION",
  "interrupted_reason": "TOKEN_LIMIT"
}
```

### 새 태스크 생성 절차

1. `harness/tasks/TASK_YYYYMMDD_NNN.md` 파일 작성
2. 큐 마커 생성: `touch harness/queue/pending/TASK_YYYYMMDD_NNN`
3. `bash harness/harness_runner.sh --task TASK_YYYYMMDD_NNN` 실행

## 6. Queue System

### 큐 상태 전환

```
pending → running → done
                 ↘ blocked  (loop_count == 3 또는 사용자 중단)
```

### 큐 디렉토리 구조

```
harness/queue/
├── pending/      ← 실행 대기 중
│   └── TASK_ID   (빈 파일 — 마커)
├── running/      ← 현재 실행 중
├── done/         ← 완료
└── blocked/      ← Safety Break 또는 사용자 중단
```

큐 마커는 내용 없는 빈 파일이다. 실제 상태는 `.state.json`에 저장된다.

### 큐 이동 규칙

- `move.sh`는 atomic `mv` 연산을 사용한다
- 소스가 존재하지 않으면 오류
- 이미 대상 위치에 마커가 있으면 오류
- BLOCKED 태스크는 `--force` 또는 `taskctl.sh resume --force` 명시 승인 없이 자동 재개 불가

### --list 출력 형식

```
ID                    STATUS   QUEUE    STEP       LOOP
TASK_20260101_001     running  running  ACTION     1
TASK_20260101_002     pending  pending  PLAN       0
TASK_20260101_003     done     done     DONE       2
```

## 7. Harness Workflow

### 전체 파이프라인

```
PLAN → RESEARCH → ACTION → REVIEW → VALIDATE → DONE
                     ↑         |         |
                     └─────────┴─────────┘ (CRITICAL/MAJOR 이슈 시 재진입)
```

### 각 단계 설명

#### PLAN
- 사용자가 태스크 `.md` 파일을 작성하여 목표, 요구사항, 검증 기준을 정의한다
- Harness는 태스크 파일을 파싱하고 초기화한다

#### RESEARCH
- Gemini CLI(`gemini -p "$PROMPT" --output-format json`)가 호출된다
- 기술 조사, 참조 문서 수집, 구현 방향 제안을 JSON으로 반환한다
- 타임아웃: `GEMINI_TIMEOUT` (기본 900초)
- 응답 파싱: Gemini 응답 envelope의 `.response` 필드 unwrap

#### ACTION
- Claude(사용자)가 RESEARCH 결과를 바탕으로 코드를 구현한다
- git branch 자동 생성: `harness/<task_id_lower>-<YYYYMMDD>` (첫 ACTION 진입 시)
- 단계 완료 후 REVIEW로 자동 전환

#### REVIEW
- Codex CLI(`codex exec --json --ephemeral -c 'sandbox_permissions=["disk-full-read-access"]'`)가 호출된다
- 정적 코드 감사: 로직 오류, 보안 취약점(OWASP Top 10), 타입 위험, CLAUDE.md 정책 위반
- 타임아웃: `CODEX_TIMEOUT` (기본 1200초)
- 응답 파싱: JSONL에서 `agent_message` 필드 추출

#### VALIDATE
- 런타임 검증: 빌드, 타입체크, 린트, 테스트, API health
- 태스크 frontmatter의 `validate:` 목록이 있으면 해당 명령만 실행
- 기본 폴백(fallback): `pnpm -r typecheck`, `pnpm -r lint`, `pnpm -r test`, `http://<TAILSCALE_IP>:8801/health`

**위험 명령 차단 목록** (VALIDATE 단계에서 실행 불가):
- `rm -rf`
- `sudo`
- `chmod -R 777`
- `git reset --hard`
- `docker system prune`

#### DONE
- Reviewer PASS + Validator PASS일 때 도달
- `--auto-commit` 또는 `HARNESS_AUTO_COMMIT=1`이면 git checkpoint 자동 생성
- `--no-chain` 없으면 다음 pending 태스크 자동 시작

### 결과 판정 로직 (`lib/policy.sh`)

```
이슈 없음                                  → CONTINUE (DONE)
이슈 등급 < severity_stop                  → PASS_WITH_WARNINGS (DONE)
이슈 등급 >= severity_stop:
  loop_count == 0                          → STOP (첫 실패 — ACTION 재진입 없음)
  0 < loop_count < retry_limit             → RETRY (ACTION 재진입)
  loop_count >= retry_limit                → STOP (한도 초과 — Safety Break)
```

### Gate 확인

각 단계 진입 전 대화형 확인 프롬프트가 표시된다:

```
[harness] Proceed to REVIEW? [y/N/s]
  y = 진행
  N = 중단
  s = 이 단계 건너뜀
```

`HARNESS_AUTO_CONFIRM=1` 또는 `--dry-run`이면 자동으로 `y`로 처리된다.

## 11. Policy & Safety

### Safety Break

`loop_count`가 3에 도달하면 자동으로 Safety Break가 발동된다:

1. `harness/logs/SAFETY_BREAK_TASKID.md` 보고서 생성
2. `harness/tasks/TASK_ID.recovery.json` 생성
3. 태스크를 `running → blocked` 큐로 이동
4. 사용자에게 보고하고 대기

Safety Break 이후 재개:
```bash
# BLOCKED 태스크 강제 재개 (사용자 명시 승인 필요)
bash harness/harness_runner.sh --resume TASK_20260101_001 --force
```

### Severity Override

MAJOR 이하 이슈를 무시하고 진행할 때 사용. CRITICAL은 항상 차단된다.

```bash
# MAJOR 이하 이슈 무시
bash harness/harness_runner.sh --task TASK_ID --override-severity MAJOR

# MINOR 이하만 무시 (MAJOR는 여전히 차단)
bash harness/harness_runner.sh --task TASK_ID --override-severity MINOR
```

**주의**: CRITICAL override는 허용되지 않는다. 코드에 하드 차단됨.

### 금지 사항 (하드 블록)

| 금지 항목 | 근거 |
|---|---|
| step 필드에 `INTERRUPTED`, `IDLE`, `DECISION` 기록 | step은 6개 enum만 허용 |
| `&&` 체이닝 없이 CLI 호출 | exit code 추적 필수 |
| `active_state.json` 무단 삭제/초기화 | 상태 손실 방지 |
| BLOCKED 태스크 자동 재개 | 사용자 명시 승인 필수 |
| checkpoint.json 없이 중단 처리 | resume 불가 방지 |

### 왜 PLAN LOOP가 필요한가

**실패 패턴** — 즉시 핫픽스 반복:

```
버그 발견
 ↓
핫픽스
 ↓
새 Root Cause 발견
 ↓
핫픽스
 ↓
재발방지 누락
 ↓
핫픽스 (반복)
```

이 패턴은 표면만 덮는다. Root Cause와 Policy GAP이 남아 있어 같은 버그가 반복된다.

**개선 패턴** — PLAN LOOP 진입:

```
버그 발견
 ↓
PLAN LOOP 진입
 ↓
Root Cause Complete (더 이상 새 원인 없음)
 ↓
Policy GAP Complete (정책 공백 없음)
 ↓
구현
 ↓
실제 E2E PASS
 ↓
Exit (Health Score 10 확인)
```

---

### Major Issue 감지 시 PLAN LOOP (Hard Constraint)

다음 조건 중 하나라도 해당하면 **즉시 핫픽스 금지**, PLAN LOOP 진입:

```
MAJOR ISSUE DETECTED → PLAN LOOP REQUIRED
```

| 조건 | 설명 |
|------|------|
| A | 이미 수정한 기능에서 새 결함 발견 |
| B | 수정 후 새 Root Cause 발견 |
| C | Contract GAP 발견 |
| D | 재발방지 정책 없음 |
| E | helper 생성 후 호출 주체 없음 |
| F | 실제 E2E 없음 |
| G | Health Score < 10 |
| H | Remaining Issues 존재 |
| I | PASS이지만 금지 문구 포함 |
| J | PASS_WITH_ISSUES 발생 |

---

### 금지 문구

다음 문구가 출력에 포함되면 자동으로 PLAN LOOP 재진입 트리거:

```
근본 해결은 아님
현재 우회
추후 구현
hook 미연결
실제 E2E 미실행
낮은 우선순위
호출 주체 없음
추후 고도화
일단 동작
임시 방편
나중에 수정
지금은 생략
TODO
FIXME (미해결)
추후 검토
실제 테스트 미실시
```

이 문구가 PASS 결과에 들어 있으면 PASS가 아니다.

---

### PASS_WITH_ISSUES 규칙

```
PASS_WITH_ISSUES는 PASS가 아니다.
PLAN LOOP 재진입 조건이다.
```

PASS_WITH_ISSUES가 발생한 경우:
1. DONE 선언 금지
2. 즉시 PLAN LOOP STEP-2(Root Cause 수집)로 재진입
3. 남은 Issues를 모두 해소한 후 다시 Exit Criteria 점검

PLAN LOOP는 10단계(재현 → Root Cause 수집 → Policy GAP → 재발방지 → 재계획 → TASK 재분할 → 구현 → E2E → Health Check → Remaining=0)를 완료해야 종료된다.

세부 정책:
- `policies/HCHAIN_MAJOR_ISSUE_DEFINITION.md` — 판정 기준 A~J
- `policies/HCHAIN_PLAN_LOOP_WORKFLOW.md` — PLAN LOOP 10단계
- `policies/HCHAIN_PLAN_LOOP_EXIT_CRITERIA.md` — 종료 조건 및 금지 문구

## 12. Troubleshooting

### 문제: 태스크가 blocked 상태에 빠짐

**원인**: loop_count가 3에 도달 (Safety Break 발동)

**조치**:
```bash
# 원인 분석
cat harness/logs/SAFETY_BREAK_TASK_ID.md
cat harness/tasks/TASK_ID.recovery.json

# 문제 수동 수정 후 강제 재개
bash harness/harness_runner.sh --resume TASK_ID --force
```

### 문제: JSON parse fail

**원인**: Gemini 또는 Codex CLI가 유효하지 않은 JSON 반환

**조치**:
```bash
# 마지막 로그 확인
ls -lt harness/logs/ | head -5
cat harness/logs/YYYYMMDD_HHMMSS_AGENT_TASKID.json | jq .
```

응답이 유효하지 않으면 `--resume`으로 해당 단계 재시도.

### 문제: Gemini/Codex CLI 인증 만료

**원인**: CLI 세션 만료 (`CLI_AUTH_EXPIRED`)

**조치**: CLI를 재인증 후 `--resume`으로 재개

### 문제: 큐 일관성 오류

**원인**: 비정상 종료로 인한 큐 마커 불일치

**조치**:
```bash
bash harness/queue/check_consistency.sh --extended
# 문제 확인 후 수동으로 큐 마커 이동
bash harness/queue/move.sh TASK_ID <wrong_queue> <correct_queue>
```

### 문제: TIMEOUT 3회 연속 발생

**원인**: CLI가 지속적으로 응답하지 않음 (네트워크 또는 CLI 문제)

**결과**: 자동으로 BLOCKED 상태로 전환

**조치**: 네트워크/CLI 상태 확인 후 `--force`로 재개

### 문제: active_state.json이 비어있거나 IDLE

**조치**:
```bash
cat harness/active_state.json | jq .
# step이 IDLE이고 task_id가 비어있으면 정상 대기 상태
# 비정상이면 큐와 state.json 확인
ls harness/queue/running/
```
