# HCHAIN 철학

## 왜 HCHAIN을 만들었는가

AI와 함께 개발하면서 반복적으로 마주친 문제가 있었다.

AI는 코드를 작성하지만, 그 코드가 **실제로 동작하는지** 보장하지 않는다. "동작합니다"라고 말하지만, 실행하면 깨진다. 로컬에서는 되는데 서버에서는 안 된다. 이번에는 고쳤는데 다음에 같은 실수가 반복된다.

더 근본적인 문제는 **절차의 부재**였다. 코드 리뷰 없이 병합하고, 테스트 없이 배포하고, 문서 없이 완료 처리했다. 사람이 직접 중계하는 구조에서는 어쩔 수 없는 일이었다.

HCHAIN은 이 경험에서 나왔다. AI 작업에 절차를 부여하고, 검증을 강제하고, 기록을 남기는 시스템이다. "AI 메신저 → AI 작업 지휘 시스템."

---

## 왜 Contract First인가

코드보다 계약이 먼저여야 하는 이유는 세 가지다.

**첫째, AI는 범위를 모른다.**
"큐에 재시도 기능을 추가해줘"라는 요청을 받으면 AI는 어디까지 수정해도 되는지 알지 못한다. DB를 바꿔도 되는지, API를 변경해도 되는지, Worker를 수정해도 되는지 — 계약 없이는 모두 추측이다.

**둘째, 영향 범위를 모르면 반드시 뭔가 빠진다.**
백엔드 기능을 추가했는데 DB 스키마 변경을 빠뜨린다. API를 바꿨는데 프론트엔드를 업데이트하지 않는다. 계약서의 `## 영향 범위`는 이 실수를 구조적으로 방지한다.

**셋째, 완료 기준이 없으면 완료를 알 수 없다.**
"동작한다"는 모호하다. 계약서의 `## 완료 기준`은 체크리스트 형태로 구체적인 완료 조건을 강제한다. 이것이 충족되어야 구현이 완료된 것이다.

---

## 왜 최소 수정인가

AI에게 자유를 주면 AI는 필요 이상으로 변경한다.

버그 하나를 고치러 들어갔다가 파일 세 개를 리팩토링하고 나온다. 기능 하나를 추가하면서 기존 인터페이스를 바꾼다. "더 나은 방법을 발견했습니다"라며 계획에 없던 일을 한다.

최소 수정 원칙은 이를 막는다. 한 번에 하나의 목적. 관련 없는 파일은 건드리지 않는다. 리팩토링과 기능 추가는 별개의 Task다.

이 원칙은 두 가지를 보장한다:
- 무엇이 바뀌었는지 명확하다
- 문제가 생기면 원인을 특정할 수 있다

---

## 왜 Reviewer가 필요한가

구현한 AI가 자신의 구현을 검토하면 편향이 생긴다.

Reviewer는 구현과 독립된 시점에서 코드를 본다. "이게 계약과 일치하는가?", "예외 처리가 빠진 곳은?", "이 코드가 기존 동작을 깨뜨리는가?" — 이 질문들을 구현자가 아닌 감사자 입장에서 던진다.

HCHAIN에서 Reviewer는 Codex CLI다. 구현 후 자동으로 정적 코드 감사를 수행하고 CRITICAL·MAJOR 이슈를 발견하면 자동으로 재작업 사이클로 진입한다. 사람이 "이거 괜찮아?"라고 물어볼 필요가 없다.

---

## 왜 Validator가 필요한가

코드가 동작하는 것과 시스템이 동작하는 것은 다르다.

Reviewer는 코드의 논리를 본다. Validator는 실제로 실행해서 확인한다. 타입체크, 린트, 테스트, API 헬스체크 — 이것들이 통과해야 진짜로 동작하는 것이다.

특히 AI 구현에서 Validator가 중요한 이유는, AI가 "동작합니다"라고 말하는 것을 믿을 수 없기 때문이다. 직접 실행해서 확인하는 것만이 신뢰할 수 있는 검증이다.

HCHAIN에서 VALIDATE 단계는 태스크 frontmatter에 정의된 셸 명령어를 실행한다. 통과해야 DONE이다.

---

## 왜 Done 파일을 남기는가

작업이 완료되었다는 기록이 없으면, 그 작업은 없었던 것이나 마찬가지다.

Done 파일은 세 가지를 기록한다:
1. **무엇을** 했는가 (변경된 파일, 추가된 기능)
2. **어떻게** 검증했는가 (테스트 결과, 검증 명령)
3. **결과가** 무엇인가 (PASS/FAIL, 남은 이슈)

이 기록은 나중에 "왜 이렇게 됐지?"라는 질문에 답한다. 버그의 원인을 추적한다. 같은 실수를 반복하지 않게 한다.

Done 없는 완료는 없다. 이것이 HCHAIN의 규칙이다.

---

## 왜 AI는 구현보다 계약이 먼저여야 하는가

사람은 코드를 작성하기 전에 머릿속에서 설계를 한다. 어떤 데이터가 필요한지, 어떤 API를 만들지, 어디서 실패할 수 있는지 — 이 생각의 과정이 있고 나서 코딩을 시작한다.

AI에게는 이 과정이 없다. 요청을 받으면 즉시 코드를 출력하기 시작한다. 생각 없이 코딩하는 개발자와 같은 문제가 생긴다 — 빠진 케이스, 잘못된 범위, 생각 없는 변경.

계약서는 AI에게 "먼저 생각해"라고 강제하는 장치다.

`--workflow`로 Contract Workflow를 실행하면 AI는 관련 계약을 읽고, 영향 범위를 분석하고, 빠진 정책을 탐지하고, 질문을 생성한다. 그러고 나서야 계약 초안을 만든다. 코드는 그 다음이다.

이 순서가 바뀌면 HCHAIN의 의미가 없다.

---

> "계약 없이 코드는 없다. 검증 없이 완료는 없다. 기록 없이 작업은 없다."
