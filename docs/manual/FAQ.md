# FAQ

## 언제 계약을 작성하나?

다음에 해당하면 계약서를 작성해야 한다:

- **신규 기능 추가** — 사용자에게 새로운 능력을 제공하는 모든 기능
- **UI 변경** — 화면 추가, 플로우 변경, 컴포넌트 구조 변경
- **API 변경** — 새 엔드포인트 추가, 기존 응답 구조 변경, 새 파라미터 추가
- **DB 변경** — 테이블 추가, 컬럼 추가, 스키마 수정, 마이그레이션
- **Queue** — 새 큐 추가, 큐 처리 로직 변경
- **Retry 정책** — 재시도 횟수, 간격, 폴백 정책 도입 또는 변경
- **State 변경** — 애플리케이션 상태 기계 변경, 새 상태 추가
- **Worker 추가** — 새 백그라운드 워커, 비동기 처리 도입
- **Scheduler 추가** — 주기적 실행 작업 추가 또는 변경
- **예외 정책 변경** — 에러 처리 방식, 알림, 로깅 정책 변경

---

## 언제 작성하지 않나?

다음은 계약서 없이 진행할 수 있다:

- **버그 수정** — 기존 계약에 명세된 동작을 올바르게 만드는 수정
- **로그 추가** — 동작 변경 없이 관찰 가능성만 높이는 작업
- **테스트 추가** — 기존 기능에 대한 테스트 케이스 작성
- **오타 수정** — 코드, 문서, 주석의 오타 교정
- **단순 리팩토링** — 외부 동작 변경 없이 내부 구조만 정리 (변수명 변경, 함수 분리 등)
- **주석 수정** — 코드 주석 업데이트

### 판단이 어려울 때

"이 변경이 기존 계약(PROJECT.md, ARCHITECTURE.md, RULES.md 등)을 위반하는가?"를 기준으로 판단한다. 위반한다면 계약서를 먼저 수정하거나 새로 작성한다.

## 왜 계약이 필요한가?

### 1. AI는 범위를 모른다

AI에게 "큐에 재시도 기능을 추가해줘"라고 말하면, AI는 어디까지 수정해도 되는지 알지 못한다. 계약서는 AI에게 정확한 범위를 알려준다.

### 2. 영향 범위를 먼저 알아야 한다

백엔드 기능을 추가했는데 DB 스키마 변경을 빠뜨리거나, API를 바꿨는데 프론트엔드를 업데이트하지 않는 문제가 반복된다. 계약서는 영향 범위를 명시하여 이를 방지한다.

### 3. 검증 기준이 없으면 완료를 알 수 없다

"동작한다"는 모호하다. 계약서의 `## 완료 기준`은 구체적이고 검증 가능한 완료 조건을 명시한다.

### 4. 기록이 남는다

계약서는 나중에 "왜 이렇게 만들었는가"를 설명하는 문서가 된다. 코드 리뷰, 버그 추적, 기능 수정 시 계약서가 맥락을 제공한다.

## Contract Workflow와 직접 계약 생성의 차이는?

**Contract Workflow** (`--workflow REQUEST`):
- 자연어 요청을 받아 6단계 파이프라인을 실행한다
- 관련 계약 자동 선택, 영향 범위 분석, 질문 생성까지 포함된다
- 새로운 기능 개발 시 권장

**직접 계약 생성** (`--generate-contract FEATURE_NAME`):
- 계약 파일 초안만 생성한다
- 영향 범위 분석은 포함되지 않는다
- 내용을 이미 알고 있을 때 빠르게 파일만 생성하려는 경우에 사용

---

## 계약서가 너무 길면 어떻게 하나?

계약서는 간결할수록 좋다. 섹션별로 최소한의 내용만 작성한다.

- `## UI` — 해당 없으면 "해당 없음"
- `## API` — 엔드포인트 목록만 (상세 명세는 별도 API 문서)
- `## 데이터` — 변경되는 필드만 (전체 스키마 아님)

---

## 계약서를 수정해도 되나?

계약서는 `draft` 상태에서 자유롭게 수정한다. `approved` 이후에는 변경 전 팀(또는 본인)의 검토가 필요하다.

라이프사이클: `draft → review → approved → implemented → deprecated`

---

## 기존 프로젝트에 HCHAIN을 적용하려면?

1. 설치 실행 (`--init-contracts`)
2. base 계약서(PROJECT.md, ARCHITECTURE.md 등) 작성
3. 기존 주요 기능에 대한 계약서 역방향 작성 (`contracts/features/`)
4. `--contract-review-diff`로 계약-코드 차이 확인
5. 이후 모든 신규 기능은 Contract First로 진행

---

## --contract-review-diff는 무엇을 분석하나?

5개 카테고리를 분석한다:

| 카테고리 | 설명 |
|----------|------|
| 계약에는 있으나 코드에 없음 | 계약서의 API·함수가 코드에 없음 |
| 코드에는 있으나 계약에 없음 | 코드의 함수가 계약서에 없음 |
| API 불일치 | 계약 API와 코드 API 이름 차이 |
| 검증 누락 | 계약서에 `## 검증` 섹션 없음 |
| 상태 정의 불일치 | 계약서의 상태(status)가 코드와 다름 |

읽기 전용 — 자동 수정하지 않는다.
