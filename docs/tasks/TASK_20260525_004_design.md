# TASK_20260525_004 Design Document
# HCHAIN Dual Agent Mode — Revised Design

Status: APPROVED
Date: 2026-05-25
Approval: APPROVED by user on 2026-05-30
Condition: agent_strategy는 HCHAIN Core 실행 필드가 아니라 Task Metadata로만 문서화한다.

---

## 1. 기존안의 문제점

### 1.1 과도하게 구체적인 Core 파라미터

기존안은 HCHAIN Core에 다음 파라미터를 도입하려 했다.

```yaml
research_type:
  - ARCHITECTURE
  - CODE
  - BOTH

review_type:
  - ARCHITECTURE
  - CODE
  - BOTH
```

이 구조의 문제:

- **Core가 특정 Agent 조합(Claude/Codex)을 알아야 한다.**
  Core는 워크플로 실행기이지 Agent 오케스트레이터가 아니다.
- **Agent 조합이 바뀌면 Core를 수정해야 한다.**
  Claude/Codex 대신 다른 Agent(Gemini, o3 등)를 쓰는 경우 Core 변경 필요.
- **RESEARCH와 REVIEW에 각각 type 파라미터가 생기면 Task 정의가 복잡해진다.**
  파라미터 조합이 많아질수록 Core 분기 로직이 증가한다.
- **Core의 책임 범위를 벗어난다.**
  Core는 흐름(PLAN→RESEARCH→ACTION→REVIEW→VALIDATE→DONE)만 관리해야 한다.

### 1.2 종속 위험 요약

| 위험 | 기존안 | 수정안 |
|------|--------|--------|
| Agent 변경 시 Core 수정 필요 | 있음 | 없음 |
| 파라미터 조합 폭발 | 있음 (type×type) | 없음 |
| Core가 Agent 이름을 알아야 함 | 있음 | 없음 |
| Skill Layer에서 해석 불가 | 있음 | 없음 |

---

## 2. 수정안

### 2.1 파라미터 구분: Core 파라미터 vs Task Metadata

두 파라미터는 성격이 다르다. Core가 해석하는 것과 사람이 읽는 것을 분리한다.

| 파라미터 | 성격 | Core 처리 여부 |
|----------|------|--------------|
| `mode` | **Core 파라미터** | Core가 읽고 동작을 변경한다 (ROOTCAUSE 시 Agent Opinion Matrix 필수) |
| `agent_strategy` | **Task Metadata** | Core는 실행 분기에 사용하지 않는다. Task 정의에 기록되며 Claude/Human이 읽는다. |

```yaml
# Core 파라미터 — harness_runner.sh가 해석한다
mode:
  - NORMAL       # 기본 실행 (기존 동작 유지)
  - ROOTCAUSE    # 이중 분석 모드: Final Report에 Agent Opinion Matrix 필수

# Task Metadata — Core는 실행 분기에 사용하지 않는다
agent_strategy:
  - DEFAULT      # 기존 동작 유지 (각 Agent 기본 설정 따름)
  - CLAUDE_ONLY  # Claude만으로 분석
  - CODEX_ONLY   # Codex만으로 분석
  - DUAL         # Claude(Architect) + Codex(Code Detective) 순서대로 실행 후 통합
```

### 2.2 Core의 역할

Core는 다음만 수행한다:

1. `mode` 값을 읽고 동작을 조정한다 (`ROOTCAUSE` 시 Final Report에 Agent Opinion Matrix 섹션 필수).
2. `agent_strategy` 필드를 Task 정의 템플릿에 포함한다 — **Core는 이 값을 실행 분기에 사용하지 않는다**.
3. PLAN → RESEARCH → ACTION → REVIEW → VALIDATE → DONE 흐름을 유지한다.

Core는 다음을 수행하지 않는다:

- Claude/Codex 자동 실행 또는 호출
- Agent별 분기 처리
- 외부 런타임 의존성 추가
- 신규 Step 생성
- 기존 Task/로그 수정

## 3. 책임 분리

### 3.1 HCHAIN Core 책임

```text
- mode 파라미터 해석 (ROOTCAUSE 시 Final Report Agent Opinion Matrix 요건 적용)
- agent_strategy 필드를 Task 템플릿에 Metadata로 포함 (실행 분기 사용 금지)
- Agent Opinion Matrix 형식 정의 (Final Report 템플릿에 포함)
- 6단계 파이프라인 무결성 유지
- 하위 호환성 보장 (미지정 시 기본값 적용)
```

### 3.2 Skill / Strategy Layer 책임

```text
- agent_strategy 값에 따른 실제 실행 지침 해석
- DUAL 전략에서 Claude/Codex 역할 정의
- ROOTCAUSE mode에서 분석 순서 및 비교 방법 정의
- Task 생성 시 적절한 strategy 추천
```

Skill Layer 정의 예시 (agent_strategy: DUAL):

```text
Claude = Architect
  - WHY 분석 (설계 의도, 구조적 타당성)
  - 아키텍처 방향 및 장기 유지보수성
  - 설계 일관성 검토

Codex = Code Detective
  - Call Graph 분석
  - Dead Code 탐지
  - 실제 실행 경로 추적
  - 중복 책임 탐지
  - 설정 누락 발견
```

### 3.3 Human Operator 책임

```text
- mode와 agent_strategy 값을 Task 작성 시 명시
- DUAL 또는 ROOTCAUSE 결과에서 최종 판단
- Agent Opinion Matrix에서 공통 결론 / 상충 결론 검토 후 결정
- 각 Agent 결과의 신뢰도 평가
```

## 4. ROOTCAUSE Mode 실행 흐름

`mode: ROOTCAUSE`인 경우 각 Step에서 다음 흐름을 따른다.

```
[RESEARCH]
  Claude
    → 구조 분석 (WHY, 아키텍처, 설계 문제)
    → 출력: Architecture Finding List

  Codex
    → 코드 분석 (HOW, Call Graph, Dead Code, 실행 경로)
    → 출력: Code Finding List

  통합
    → 두 Finding List 비교
    → 공통 결론 추출
    → 상충 결론 기록

[REVIEW]
  Claude
    → Architecture Review (구조적 타당성, 확장성, 유지보수성)
    → 출력: Architecture Review Notes

  Codex
    → Code Review (회귀 위험, Side Effect, 숨겨진 의존성)
    → 출력: Code Review Notes

  통합
    → 두 Review 결과 병합
    → Agent Opinion Matrix 생성

[DONE]
  Final Report에 Agent Opinion Matrix 포함 (필수)
```

## 5. Agent Opinion Matrix 형식

`mode: ROOTCAUSE` 또는 `agent_strategy: DUAL` 인 경우
Final Report 말미에 다음 형식을 포함한다.

```markdown
## Agent Opinion Matrix

| 항목 | Claude 결론 | Codex 결론 | 공통 결론 | 상충 여부 |
|------|------------|-----------|----------|----------|
| [분석 항목 1] | [Claude 분석 결과] | [Codex 분석 결과] | [합의된 결론] | 없음 / 있음 |
| [분석 항목 2] | ... | ... | ... | ... |

### 공통 결론 요약
- [두 Agent가 동의한 핵심 발견]

### 상충 결론 (있는 경우만)
- [Claude 주장] vs [Codex 주장]
- 해결 방안: [Human Operator 판단 필요]
```

실제 예시:

```markdown
## Agent Opinion Matrix

| 항목 | Claude 결론 | Codex 결론 | 공통 결론 | 상충 여부 |
|------|------------|-----------|----------|----------|
| Audit 신뢰도 | Audit 구조가 실제 화면 검증 미수행 | Caption Owner가 2개 존재 | Audit 신뢰도 부족 | 없음 |
| 데이터 정합성 | 설계상 Owner 단일화 필요 | DB에 중복 Owner 레코드 확인 | Owner 중복이 Audit 실패 원인 | 없음 |

### 공통 결론 요약
- Audit 신뢰도가 낮은 근본 원인은 Caption Owner 중복이다.

### 상충 결론
- 없음
```

## 6. 하위 호환성

### 6.1 기본값 정의

| 파라미터 | 미지정 시 기본값 | 동작 |
|---------|----------------|------|
| `mode` | `NORMAL` | 기존 동작 유지 |
| `agent_strategy` | `DEFAULT` | 기존 동작 유지 (Skill 기본 설정 따름) |

### 6.2 기존 Task 영향 없음

- `mode`, `agent_strategy` 필드가 없는 기존 Task는 `NORMAL` + `DEFAULT`로 동작한다.
- 기존 Task 파일을 수정하지 않는다.
- 기존 harness 실행 경로는 변경하지 않는다.

### 6.3 점진적 도입 가능

- 새로운 Task에서만 선택적으로 `mode: ROOTCAUSE`를 사용할 수 있다.
- 기존 파이프라인과 병행 운용 가능하다.

## 7. 금지 사항

이 설계를 구현할 때 다음은 절대 금지한다.

```text
❌ 신규 Step 추가 금지
   - RESEARCH_ARCH, RESEARCH_CODE, REVIEW_ARCH, REVIEW_CODE 등 별도 Step 추가 금지

❌ install.sh 수정 금지

❌ harness_runner.sh 수정 금지

❌ 자동 Agent 실행 트리거 금지
   - Core가 Claude/Codex를 자동 호출하는 로직 추가 금지

❌ 기존 Task/로그 수정 금지
   - 기존 tasks/, logs/, queue/ 내 파일 수정 금지

❌ 외부 런타임 의존성 추가 금지
   - Python, Node.js, Ruby 등 외부 런타임 추가 금지

❌ API 키 또는 환경 변수 참조 추가 금지

❌ 여러 파일 동시 수정 금지
   - 승인된 범위의 파일만, 하나씩 수정
```
