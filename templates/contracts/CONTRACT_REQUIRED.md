# 계약 필요 여부 판단 기준

## 계약이 필요한 경우

다음 중 하나에 해당하면 반드시 `contracts/features/` 에 기능 계약서를 작성한다.

### 기능·구조 변경

- **신규 기능** — 사용자에게 새로운 능력을 제공하는 기능 추가
- **UI 변경** — 화면 추가, 사용자 플로우 변경, 컴포넌트 구조 변경
- **API 변경** — 새 엔드포인트 추가, 기존 응답 구조 변경, 새 파라미터 추가

### 데이터·상태 변경

- **DB 변경** — 테이블 추가, 컬럼 추가, 스키마 수정, 마이그레이션
- **State 변경** — 애플리케이션 상태 기계 변경, 새 상태 추가

### 비동기·스케줄 처리

- **Queue** — 새 큐 추가, 큐 처리 로직 변경
- **Retry 정책** — 재시도 횟수·간격·폴백 정책 도입 또는 변경
- **Worker 추가** — 새 백그라운드 워커, 비동기 처리 도입
- **Scheduler 추가** — 주기적 실행 작업 추가 또는 변경

### 정책 변경

- **예외 정책 변경** — 에러 처리 방식, 알림, 로깅 정책 변경

---

## 계약이 필요하지 않은 경우

다음에 해당하면 계약서 없이 진행할 수 있다.

- **버그 수정** — 기존 계약에 명세된 동작을 올바르게 만드는 수정
- **로그 추가** — 동작 변경 없이 관찰 가능성만 높이는 작업
- **테스트 추가** — 기존 기능에 대한 테스트 케이스 작성
- **오타 수정** — 코드, 문서, 주석의 오타 교정
- **단순 리팩토링** — 외부 동작 변경 없이 내부 구조만 정리
- **주석 수정** — 코드 주석 업데이트

---

## 판단 기준

> 이 변경이 기존 계약(PROJECT.md, ARCHITECTURE.md, RULES.md 등)을 위반하는가?

위반한다면:
1. 기존 계약서를 수정하거나
2. 새 기능 계약서를 작성한다

위반하지 않고 외부 동작도 바뀌지 않는다면 계약서 없이 진행한다.

---

## 계약 작성 방법

```bash
# Contract Workflow로 자동 생성 (권장)
python3 ~/hchain/install.py --target . --workflow "기능 요청 내용"

# 직접 생성
python3 ~/hchain/install.py --target . --generate-contract FEATURE_NAME
```

자세한 내용은 `docs/manual/FEATURE_CONTRACT.md` 참조.
